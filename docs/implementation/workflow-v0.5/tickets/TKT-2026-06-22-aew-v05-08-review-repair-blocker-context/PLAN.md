## Review And Repair Model

- Independent reviewers inspect actual diff, changed files, tests, evidence, ticket scope, and spec alignment.
- `FAIL` must include bounded findings with file/path, expected behavior, actual behavior, reproduction or proof command, failed acceptance criterion, and required repair.
- Repair uses a fresh writer, preserves acceptance criteria and tests, and returns to independent review.
- Repair loop cap is three materially different attempts.
- After the cap, create a blocker capsule instead of silently completing.
- A ticket cannot be marked done unless the latest independent review after all repairs is `PASS`.

## Blocker Resolver Model

The blocker resolver receives only a minimal facts capsule, not raw transcript. It returns classification, confidence, smallest safe action, exact allowed commands or edits, validation, rollback, and whether one user question is required.

Policy matrix:

- focused in-scope repair: preapproved,
- dependency install: preapproved only with accepted plan, existing package manager and lockfile, reviewer acceptance, and passing checks,
- reversible local/test migration: preapproved only for disposable or test state,
- cleanup: preapproved only for artifacts generated by the active run,
- commit/push: preapproved only when assigned by ticket delivery with scoped staging and no force push,
- secrets, production, deploy/publish, remote history rewrite, destructive user-work cleanup, and irreducible product/policy choices: user-required.

## Context Curation Model

The context curator runs only after review `PASS`. It records durable facts only: ticket id and phase, decisions, changed workflow contracts, proof summaries, risks, follow-ups, scoped paths, and next action. It excludes raw logs, raw transcripts, secrets, `.env*`, forbidden paths, unrelated local paths, speculative notes, and unrelated context.
